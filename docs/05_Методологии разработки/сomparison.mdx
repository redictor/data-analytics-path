# Чем SCRUM отличается от Kanban?

SCRUM и Kanban - это две популярные гибкие методологии управления проектами, которые относятся к Agile-подходу. Обе направлены на повышение эффективности и гибкости, но имеют существенные различия в структуре, принципах и применении. Для примера, самое очевидное - это различие между исчислением сроков проекта и его дедлайнов.

| Параметр сравнения | SCRUM | Kanban |
|--------------------|-------|--------|
| **Структура и циклы работы** | - Работа организуется в спринты  <br /> - Каждый спринт начинается с планирования и заканчивается ретроспективой  <br /> - В конце спринта команда должна предоставить готовый продукт или его часть | - Работа выполняется в непрерывном потоке без жёстких временных рамок  <br /> - Нет спринтов, задачи добавляются и завершаются по мере необходимости  <br /> - Акцент на постоянном улучшении процесса |
| **Роли и ответственность** | - Чётко определённые роли: Владелец продукта (Product Owner), Скрам-мастер (Scrum Master) и Команда разработки  <br /> - Ответственность за выполнение задач строго распределена | - Нет предопределённых ролей  <br /> - Команда работает как единое целое  <br /> - Ответственность распределяется более гибко |
| **Планирование** | - Планирование происходит в начале каждого спринта  <br /> - Команда выбирает задачи из бэклога продукта  <br /> - Задачи фиксируются на весь спринт (изменения в процессе спринта не приветствуются) | - Планирование более гибкое  <br /> - Задачи добавляются в процесс по мере необходимости  <br /> - Изменения могут вноситься в любой момент |
| **Метрики** | - Основная метрика - скорость команды (Velocity), которая измеряет, сколько задач команда может выполнить за спринт | - Основные метрики - время цикла (Cycle Time) и время выполнения (Lead Time), которые показывают, сколько времени занимает выполнение задачи |
| **Визуализация работы** | - Используется спринт-доска (по факту адаптированная kanban-доска), которая обновляется в начале каждого спринта  <br /> - Доска обычно включает столбцы: "Сделать", "В процессе", "Готово" | - Используется Kanban-доска, которая отображает весь рабочий процесс  <br /> - Столбцы соответствуют этапам процесса, а задачи перемещаются по ним по мере выполнения  <br /> - Акцент на ограничении количества задач в процессе (WIP - Work In Progress) |
| **Гибкость** | - Более структурированный подход  <br /> - Изменения в рамках спринта не приветствуются  <br /> - Подходит для проектов с чёткими целями и сроками | - Более гибкий подход  <br /> - Изменения можно вносить в любой момент  <br /> - Подходит для процессов с постоянно меняющимися приоритетами |
| **Цели** | - Нацелен на выполнение конкретных задач в рамках спринта и предоставление готового продукта | - Нацелен на оптимизацию потока работы и снижение времени выполнения задач |
| **Контроль нагрузки** | - Нагрузка определяется на этапе планирования спринта  <br /> - Команда заранее оценивает, сколько задач сможет взять в работу | - Используются WIP-лимиты  <br /> - Ограничивается количество задач, которые могут одновременно находиться в работе |
| **Подход к изменениям** | - Изменения стараются переносить на следующий спринт  <br /> - Внутри активного спринта стабильность важнее гибкости | - Изменения допускаются сразу  <br /> - Процесс легко адаптируется под новые условия и приоритеты |
| **Подходит для** | - Разработки новых продуктов  <br /> - Команд с фиксированным составом  <br /> - Проектов, где важны регулярные итоги и спринтовая дисциплина | - Поддержки, сервисных команд, аналитики, DevOps  <br /> - Процессов с постоянным потоком входящих задач  <br /> - Ситуаций, где приоритеты часто меняются |
| **Встречи и ритуалы** | - Есть обязательные события: планирование спринта, daily stand-up, review, retrospective | - Обязательных ритуалов меньше  <br /> - Встречи могут использоваться по необходимости |
| **Предсказуемость сроков** | - Выше за счёт фиксированных спринтов и заранее выбранного объёма задач | - Ниже в классическом виде, но лучше видно скорость потока и узкие места процесса |
